Enhance PulseDesk (multi-tenant SaaS help desk at pulsedesk.support) with a fully functional, production-minded “Email-to-Ticket” system that is available ONLY to tenants on the Enterprise or Unlimited subscription plans.

PROJECT GOAL
Build a secure, tenant-isolated inbound email processing feature where each qualified tenant gets a dedicated inbound support email identity that automatically creates tickets in that tenant’s workspace when emails are received. This must be a real backend feature, not a front-end simulation.

CORE BUSINESS RULES
- PulseDesk is multi-tenant.
- Only tenants with subscription plan = "enterprise" or "unlimited" can access and use Email-to-Ticket.
- Lower plans must not have access to the feature in UI or backend.
- If a tenant downgrades below Enterprise, inbound email-to-ticket creation must be disabled immediately for future inbound messages.
- Historical tickets remain intact after downgrade, but new incoming emails must no longer create tickets.
- Access to this feature must be derived from a proper entitlement or feature flag system, not hardcoded UI checks.

TENANT-SPECIFIC INBOUND EMAIL DESIGN
Implement a scalable inbound email design using a tenant-specific alias such as:
- support+tenant-slug@pulsedesk.support
or an equivalent clean routing model.

Requirements:
- Each Enterprise or Unlimited tenant gets a unique inbound support email alias.
- Store the alias in the database.
- Prevent alias collisions.
- Tenant admins can view and copy their assigned address from settings.
- Alias regeneration should be restricted to super admin or protected by a strong confirmation warning because it affects routing.
- Do NOT create one separate real mailbox per tenant. Use a scalable alias-routing model instead.

PLAN GATING / ENTITLEMENTS
Implement a robust feature entitlement system for billing plans:
- enterprise and unlimited => email_to_ticket = true
- all lower plans => email_to_ticket = false
- Enforce this in backend services, API routes, and middleware.
- UI should show Email-to-Ticket settings only to eligible tenants.
- Lower plans should see a locked premium feature card with upgrade messaging.
- Unauthorized tenants must not be able to enable or invoke the feature through manual requests or API tampering.

EMAIL INGESTION ARCHITECTURE
Create a provider-agnostic inbound email processing system with an adapter abstraction so the platform can later connect to:
- SendGrid Inbound Parse
- Mailgun Routes
- AWS SES inbound
- Postmark inbound

Build the system so provider implementation can be swapped later without major rewrites.

Required processing flow:
1. Receive inbound webhook payload or normalized parsed email object
2. Identify target tenant from destination alias/address
3. Verify tenant exists and entitlement is active
4. Parse sender name, sender email, recipient alias, subject, plain text body, HTML body, message-id, in-reply-to, references, headers, attachments, timestamps, SPF/DKIM/DMARC metadata if available
5. Determine whether the message belongs to an existing thread or should create a new ticket
6. Create ticket or append reply accordingly
7. Log the inbound event and outcome

TICKET CREATION LOGIC
When a new inbound email is received and no matching thread exists:
- Create a ticket inside the correct tenant only
- Title = subject line
- Source = "email"
- Default status = "open"
- Default priority = "normal" unless tenant rules say otherwise
- Create requester contact from sender email if no matching contact exists in that tenant
- Store original email metadata for audit/history
- Build initial description from cleaned plain text body
- Strip signatures, excessive quote chains, and obvious reply history where reasonably possible
- Optionally map to queue/department/team if tenant settings define a default

THREADING / REPLY MATCHING
Implement proper email reply threading using:
- Message-ID
- In-Reply-To
- References
- optional subject token support like [PD-1234]

Requirements:
- If inbound mail matches an existing ticket thread, append it as a ticket reply/comment instead of creating a duplicate ticket
- Preserve sender identity and original email metadata
- Maintain full audit history of the inbound message
- Do NOT rely on subject line alone for threading

SECURITY / ABUSE PROTECTION
Implement strong protections:
- Validate webhook authenticity/signatures where provider supports it
- Sanitize all email content before storage/rendering
- Never render raw HTML without sanitization
- Prevent XSS via HTML email bodies
- Validate attachments by size and extension
- Add a placeholder hook/service for malware scanning
- Add rate limiting / abuse controls on inbound endpoints
- Reject mail to invalid, disabled, or unauthorized aliases
- Prevent cross-tenant routing under all circumstances
- Support optional allowed sender domains / allowlist per tenant
- Surface spoofing risk flags when SPF/DKIM/DMARC results are present
- Log all accepted, rejected, failed, threaded, and created events with reason codes

Include proper indexes on:

Use clean relational design with strong tenant scoping on all relevant records.
