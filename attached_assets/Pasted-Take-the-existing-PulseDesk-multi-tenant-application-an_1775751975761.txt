Take the existing PulseDesk multi-tenant application and enhance the Microsoft 365 / Entra integration experience with ADMIN CONSENT GUIDANCE, LIVE SIGN-IN TEST FLOW SUPPORT, GROUP LOOKUP / ROLE MAPPING ASSISTANCE, and a CLEAN ONBOARDING COMPLETION FLOW.

IMPORTANT:
This is not a redesign.
This is a final Microsoft 365 onboarding and integration-hardening pass that makes the tenant admin experience feel operationally complete and easier to deploy across many future healthcare tenants.

PRIMARY GOAL:
Improve PulseDesk’s tenant-scoped Microsoft 365 / Entra onboarding flow so a tenant admin can move from configuration to validation to real test sign-in to go-live readiness with much less friction.

SECONDARY GOAL:
Reduce implementation ambiguity and support burden by giving tenant admins better tools, better guided flows, and clearer setup completion signals.

MULTI-TENANT SAFETY REQUIREMENTS:
- all behavior must remain tenant-scoped
- all admin consent guidance, tests, group mapping assistance, results, and onboarding completion states must remain tenant-scoped
- one tenant must never see another tenant’s Entra config, test users, role mappings, group lookup data, or readiness state

MAIN FEATURES TO BUILD:
1. Admin Consent Guidance Experience
2. Live Microsoft Sign-In Test Flow
3. Group Lookup / Mapping Assistance
4. Onboarding Completion Flow
5. Go-Live Readiness Confirmation
6. Tenant Admin Guidance UX Improvements

ADMIN CONSENT GUIDANCE REQUIREMENTS:
Add a tenant-admin-facing guidance section that helps explain and support Microsoft 365 / Entra admin consent where applicable.

The UI should:
- explain when admin consent may be required
- explain why tenant admin approval may be needed
- provide a clean, productized explanation rather than raw developer jargon
- support a copy-friendly or clickable admin consent link if the architecture supports it
- show whether consent appears pending, completed, unknown, or not required
- include helpful guidance for next steps if consent has not yet been completed

Do NOT expose raw OAuth/OIDC jargon unnecessarily to standard tenant admins.
Keep this professional and implementation-friendly.

LIVE SIGN-IN TEST FLOW REQUIREMENTS:
Add a tenant-scoped “Test Microsoft Sign-In” workflow that simulates or performs a real Microsoft sign-in flow in a safe and guided way.

The test flow should:
- start from the tenant admin setup area
- clearly indicate this is a validation test
- use the active tenant’s Microsoft 365 configuration
- confirm whether sign-in succeeded
- confirm whether the returned identity belongs to the expected tenant
- confirm whether PulseDesk role mapping succeeded
- confirm whether the user would be authorized or denied
- capture a clean summary of test outcome
- store tenant-scoped test history or at least last successful test info

Test result states should include examples like:
- Sign-In Test Passed
- Sign-In Test Passed with Warnings
- Sign-In Test Failed
- Sign-In Succeeded but No Role Mapping Found
- Sign-In Succeeded but User Not Authorized
- Sign-In Succeeded but Tenant Mismatch Detected
- Consent Required
- Configuration Incomplete

For the test results, show:
- timestamp
- tested user display name if appropriate
- auth source
- mapped role result
- authorization result
- recommended next action

GROUP LOOKUP / MAPPING ASSISTANCE REQUIREMENTS:
Improve the tenant admin role-mapping experience by helping admins identify or verify Microsoft Entra groups.

If possible within architecture and permissions, add support for one or more of the following:
- manual entry of group object IDs
- friendly display labels for mapped groups
- group lookup helper placeholder
- import/verify group IDs if available later
- mapping verification checks

At minimum:
- support storing stable group object IDs as the canonical mapping value
- support optional friendly labels shown in the UI
- validate whether role mappings appear complete
- warn if no Admin role mapping exists
- warn if duplicate mappings or conflicting mappings exist
- explain why object IDs are more reliable than display names

Do not hardcode group names.
Do not assume all tenants use the same naming pattern.
